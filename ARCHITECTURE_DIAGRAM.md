```
                      KEY ARCHITECTURAL DECISIONS
â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•â•

1. MONOLITHIC ARCHITECTURE (Current)
   Rationale: Simpler deployment, lower cost, suitable for MVP
   Trade-off: Harder to scale individual components

2. STATELESS BACKEND
   Rationale: Easy horizontal scaling, no session storage needed
   Trade-off: Auth state managed client-side

3. FIRESTORE OVER SQL
   Rationale: NoSQL flexibility, automatic scaling, real-time updates
   Trade-off: Limited complex query capabilities

4. CLIENT-SIDE RENDERING (CSR)
   Rationale: Rich interactivity, separation of concerns
   Trade-off: SEO challenges, initial load time

5. GEMINI AI INTEGRATION
   Rationale: State-of-the-art NLP, cost-effective, multimodal capable
   Trade-off: Vendor lock-in, API rate limits

6. PDF TEXT EXTRACTION (Server-side)
   Rationale: Better security, consistent processing
   Trade-off: Server resource usage

7. RENDER PLATFORM
   Rationale: Simple deployment, free tier, GitHub integration
   Trade-off: Cold starts on free tier

8. ACTIVITY TRACKING (LocalStorage)
   Rationale: No database load, instant updates, privacy-friendly
   Trade-off: Data lost on cache clear, not synced across devices
```
